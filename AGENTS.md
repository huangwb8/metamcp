# MetaMCP Fork - 项目指令

本仓库基于 MetaMCP 源码进行二次独立开发，用于持续演进为你自己的产品与工具链。默认假设：当前工作的目标是服务这个 fork 的真实需求，而不是整理一套面向上游仓库的 PR。

## 项目目标

- 保留并演进 MetaMCP 的核心能力：MCP 聚合、代理、命名空间、中间件、认证、管理后台与文档。
- 优先优化当前 fork 的可维护性、可部署性、可调试性与开发效率。
- 允许按实际需求调整命名、目录、接口边界与产品策略；只有在明确有收益时才考虑向上游兼容。
- 当后端协议、共享类型或 UI 行为发生变化时，确保前后端与共享包一起完成闭环修改。

## 项目结构

- `apps/backend`：Express 5 后端，提供 `/api/auth`、`/metamcp`、`/mcp-proxy`、`/trpc` 等入口；`src/lib/metamcp/` 是 MCP 聚合、会话池和中间件核心。
- `apps/frontend`：Next.js 15 管理界面，主要页面位于 `app/[locale]/(sidebar)/`，包含 MCP servers、namespaces、endpoints、search、live logs 等管理入口。
- `packages/trpc`：前后端共享的 tRPC 路由定义与类型出口。
- `packages/zod-types`：共享的 Zod schema 与协议数据结构。
- `packages/eslint-config`、`packages/typescript-config`：仓库级开发规范配置。
- `docs`：文档站内容、图片与多语言文档页面。
- 根目录：`turbo.json`、`pnpm-workspace.yaml`、`docker-compose*.yml`、`example.env` 等负责 monorepo 编排、开发环境和部署入口。

## 常用命令

- 安装依赖：`pnpm install`
- 本地联调：`pnpm dev`
- Docker 开发环境：`pnpm run dev:docker`
- 全局构建：`pnpm build`
- 全局检查：`pnpm lint`、`pnpm check-types`
- 后端测试：`pnpm --filter backend test`
- 后端数据库：`pnpm --filter backend db:generate:dev`、`pnpm --filter backend db:migrate:dev`
- 前端单独检查：`pnpm --filter frontend check-types`、`pnpm --filter frontend build`

## 核心工作流

当用户提出与本仓库相关的需求时，按以下流程执行：

### 1. 任务理解

- 先判断改动属于后端、前端、共享包、文档还是部署配置。
- 明确该需求是为了当前 fork 的产品目标，还是仅仅在延续上游习惯；优先满足当前 fork。
- 识别是否涉及跨包联动，例如后端接口变化通常会影响 `packages/zod-types`、`packages/trpc` 与前端调用。

### 2. 执行流程

- 先阅读受影响模块的入口文件与相邻实现，再动手修改。
- 优先做最小但完整的端到端改动，不留下半同步状态。
- 如果涉及 MCP 聚合逻辑、tool schema、namespace/endpoint 行为，额外检查后端路由、共享 schema、前端页面与文档是否需要一起更新。
- 如果涉及环境变量、端口、认证或部署方式，同时检查 `example.env`、Docker 配置和相关说明文档。

### 3. 验证流程

- 小改动至少运行受影响范围内的类型检查或测试。
- 后端逻辑改动优先运行：`pnpm --filter backend test`
- 前端页面或共享类型改动优先运行：`pnpm --filter frontend check-types`
- 跨包改动优先运行：`pnpm check-types`
- 只有在任务确实要求或改动范围足够大时，再补跑 `pnpm build` 或全局检查。

### 4. 输出规范

- 先说明用户会感知到的结果，再补充验证与剩余风险。
- 不把输出写成面向上游维护者的 PR 描述，除非用户明确要求。
- 文件引用、命令与后续建议尽量具体，方便直接继续开发。

## Fork 开发策略

- 默认不以向上游仓库提交 PR 为目标。
- 不必为了“保持最小 diff”而拒绝合理的重命名、重构或模块边界调整。
- 不必主动维护上游贡献者口径的文案、流程或提交习惯，除非它们对当前 fork 仍然有价值。
- 如果某项大改动会显著提高未来同步上游安全修复的成本，需要在实施前明确提醒这个权衡。

## 工程原则

本项目遵循以下工程原则：

| 原则 | 核心思想 | 在本项目中的体现 |
|------|----------|------------------|
| **Fork First** | 先服务当前 fork 的目标 | 允许为了当前产品方向重构目录、命名与模块边界 |
| **KISS** | Keep It Simple, Stupid | 优先最直接可维护的实现，避免为假想需求堆抽象 |
| **YAGNI** | You Aren't Gonna Need It | 只实现当前确定需要的能力 |
| **DRY** | Don't Repeat Yourself | 后端、前端与共享类型中的重复协议应统一收敛 |
| **SoC** | Separation of Concerns | 路由、业务逻辑、schema、UI 状态与部署配置分层清晰 |
| **Type Safety** | 类型先行 | 共享结构尽量经由 `zod` 和 `trpc` 保持一致 |
| **Least Astonishment** | 最小惊讶原则 | API、页面行为与配置命名应符合直觉 |
| **Early Return** | 尽早返回 | 降低嵌套层级，提升可读性 |

原则冲突时的优先级：
1. 正确性
2. 可维护性
3. 清晰性
4. 与上游兼容性

## 默认语言

除非用户明确要求其他语言，默认使用简体中文进行交流、注释说明与新增文档。

## 联网与搜索

默认优先使用项目内文件、本地命令与现有文档建立结论。只有在本地信息不足、问题明显依赖外部最新事实时，才补充联网检索，并保留关键来源。

## Codex CLI 特定说明

### 文件引用规范

- 优先引用具体文件与起始行号。
- 不要把刚生成的大段文件内容直接贴回对话，除非用户明确要求。
- 涉及多个文件时，先给最关键的入口文件，再补充相关文件。

### 代码编辑规范

- 修改前先读取足够上下文，避免只改表面调用点。
- 优先做逻辑成组的修改，减少来回抖动式编辑。
- 不要顺手做与任务无关的大规模格式化或“清理”。
- 发现用户自己的未提交改动时，默认保留并在其基础上继续工作。

### 输出格式

- 简单任务用简洁结论即可。
- 复杂任务优先总结结果、验证和风险，不写成流水账。
- 如果没能完成某项验证，要明确说明原因。

## 变更边界

- 仅修改与当前任务直接相关的文件，但跨包一致性所必需的联动修改不算越界。
- 默认不为上游 PR 额外打磨提交粒度、文案或兼容层。
- 不覆盖用户本地的密钥、环境变量或无关配置。
- 对部署、认证、数据库或公共协议的变更，要显式检查下游影响。

## 变更记录规范

凡是项目的更新，都要统一记录到 `CHANGELOG.md`。

需要记录的典型情况：
- `AGENTS.md`、`CLAUDE.md` 的修改
- 目录结构或关键文件调整
- 核心工作流或开发约定变化
- 影响行为的配置、依赖或接口变更
- 用户可感知的功能变化或修复

记录要求：
- 写清改了什么、为什么改、影响了哪里
- 优先记录真实的用户或开发者影响
- 发布前将 `[Unreleased]` 的内容整理到正式版本条目

## 版本管理

当前仓库是 `pnpm` monorepo，版本信息并没有统一收敛到单一配置文件：

- 根目录 `package.json` 是私有工作区配置，不承担发布版本管理职责。
- 各子包如有版本号，以各自 `package.json` 为准。
- 仓库级的可追溯变更，以 `CHANGELOG.md` 和 git tag 为主。

如果未来要引入统一版本号来源，先明确 Single Source of Truth，再同步到文档、包配置与发布流程。

## 有机更新原则

更新本文档时：

- 先确认变化是在服务当前 fork 的什么目标。
- 再判断这条规则应该落在哪个章节，避免把文档写成松散堆叠。
- 如果更新了工作流、结构或边界，同时同步 `CLAUDE.md` 与 `CHANGELOG.md`。
- 当某个章节开始失真、过时或过长时，主动重写而不是继续打补丁。

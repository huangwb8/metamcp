This fork supports automatic Docker Hub publishing for release images.

## Publishing flow

1. Create and publish a GitHub Release for the repository.
2. GitHub Actions workflow `publish-release-images` resolves the release tag.
3. The workflow checks Docker Hub first and only builds missing tags.
4. If tags are missing, it builds the root `Dockerfile` for `linux/amd64` and `linux/arm64`.
5. It pushes the release tags to Docker Hub and keeps `latest` aligned with stable releases only.

## Workflow triggers

- Immediately when a GitHub Release is published
- Manually through GitHub Actions `workflow_dispatch`
- Every 12 hours as a backfill/self-healing check

## Required repository configuration

Repository Variables:

- `DOCKERHUB_IMAGE`: optional full image repository override such as `my-org/metamcp` (repository only, no `:tag`)
- `DOCKERHUB_NAMESPACE`: required when `DOCKERHUB_IMAGE` is not set
- `DOCKERHUB_REPOSITORY`: optional repository override; defaults to the GitHub repository name

Repository Secrets:

- `DOCKERHUB_USERNAME`: Docker Hub username
- `DOCKERHUB_TOKEN`: Docker Hub access token with push permission

## Tag strategy

For a stable release like `v1.2.3`, the workflow publishes:

- `v1.2.3`
- `1.2.3`
- `1.2`
- `1`
- `latest`

For pre-releases, the workflow publishes only the release-specific tags and does not move `latest`.

## Local deployment

`docker-compose.yml` keeps GHCR as the default image source for backward compatibility. To deploy from Docker Hub instead, set `METAMCP_IMAGE` in `.env`:

```bash
METAMCP_IMAGE=docker.io/<your-dockerhub-namespace>/metamcp:latest
```

That lets operators switch registries without editing the compose file itself.
